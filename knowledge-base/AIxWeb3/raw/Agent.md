## Notes

* An Agent is an AI system that can continuously call tools, read state, and adjust steps around a goal.
* It can move from suggestion to action inside clear permissions and an auditable process.
* That is also the real risk point of Agents: errors are no longer just "wrong answers"; they become "wrong actions."
	* **Tools are more dangerous than answers**
* So the core of an Agent design is **not making the model more human-like**, but **giving the execution loop clear boundaries**
	* It is important to establish division of labor:
		* **the model proposes candidate actions
		* **the system limits the action space**
		* **the user approves high-risk boundaries
* **An Agent is not autonomy itself; it is a constrained execution loop** with:
	* **Goal, tools, state, permissions, and stop conditions**
	* The most dangerous design is giving an Agent vague goals, broad tools, long-term memory, and large-asset permissions at the same time.
- A usable Agent must know:
	- what it can do / what it cannot do
	- how to verify completion
	- how to stop on failure
	- who can audit what it has done
- **Agent State must be externalized and queryable - not hidden in context**
- **Stop conditions must be explicit**
	- e.g. reaching the goal, exceeding budget, lacking information, crossing risk boundaries, or user rejection should all stop the Agent
- **Concepts**
	- **Tool Use**: calls external capabilities, turning an Agent from "can answer" into "can do."
		- Tools also make Prompt Injection and model misjudgment dangerous
			- e.g. an agent influenced by a malicious webpage
		- **Tool design** SHOULD make these clear:
			- input schema
			- permission scope
			- whether it is read-only
			- whether it creates external side effects
			- how calls are logged before and after execution
			- which calls require human confirmation
	- **Planning**: breaks a goal into steps
		- A plan generated by the model is only a candidate route, not authorization
		- The closer it gets to high-risk action, **the more the plan must be broken apart and checked by system rules.**
		- **A good Agent plan** SHOULD expose:
			- which tool each step needs
			- whether each step reads or writes
			- which steps can run automatically
			- which steps require user confirmation
			- whether failure can be retried
			- how final task completion will be verified
	- **State**: is the current task state of the agent.
		- State includes: user goal, completed steps, tool returns, errors, budget, confirmation records, and final output.
			- In scenarios with external execution, state should also record environment, version, key parameters, tool-call results, confirmation requests, and revocation events.
		- **Keeping state only in prompt history is not enough**
		- Production systems need queryable, recoverable, auditable state
	- **Reflection**: is a process that asks the Agent to check its intermediate results (failure reasons, logs, etc) to then correct the next step
		- Reflection improves the quality of complex tasks but it can only assist in diagnosis; it cannot be the final safety judgment.
		- **Self-checking can improve quality; deterministic checking is what can carry risk.**
	- **Multi-agent**: several Agents divide work in complex workflows
		- Amplifies coordination problems
			- lost context transfer, unclear responsibility boundaries, treating another Agent's mistake as fact, and tool permissions spreading across roles
		- If you only split one unclear process into multiple unclear roles, the system becomes harder to debug.
- **Where It Fits in AI x Web3**
	- **Agents sit between model capability and on-chain execution**
	- A relatively stable AI x Web3 Agent architecture usually looks like this:
		1. The user provides a goal and constraints.
		2. The Agent reads context and generates a plan.
		3. The system splits the plan into read-only steps and candidate write steps.
		4. Read-only tools run automatically; write tools enter policy checks.
		5. Simulation shows on-chain impact.
		6. The user confirms high-risk actions.
		7. Wallet / Smart Account executes.
		8. Logs record each step and final state.
