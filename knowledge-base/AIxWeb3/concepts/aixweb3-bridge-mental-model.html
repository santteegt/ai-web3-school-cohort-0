AI × Web3 Bridge — System Mental Model

8 core modules that every AI × Web3 agent system must account for — how they connect, which standards implement them, and the one design principle that matters most per module.

Key Design Principles — One per Module

① Data Layer
Chain-aware Context
Re-verify at execution time — context becomes stale with every block. Never trust a context snapshot from context-building time to still be valid when the transaction fires.
JSON-RPC · The Graph · Block Explorer API
② Tooling Layer
Web3 Tool Use
Read/write separation is mandatory. Reading on-chain state and sending transactions must be different tools with different permission levels — irreversible writes cannot share access grants with safe reads.
MCP · JSON-RPC · ERC-7231
③ Orchestration
Agent Workflow
Not all steps should be automated — high-risk, irreversible, or compliance-sensitive steps require human-in-the-loop. Automation is not binary; it's a per-step risk decision with explicit stop conditions.
LangGraph · FSM · A2A Protocol
④ Execution
Agent Wallet
Never give agents primary wallet keys. Agents must receive delegated permissions that are scoped (specific contracts/methods/amounts), time-limited (session keys), and revocable by the user at any time.
ERC-4337 · Session Keys · EIP-7702
⑤ Financial
Machine Payment
Budget precedes execution — there is no safe autopayment without layered spending limits (global, task, per-call, emergency stop). Decouple payment intent from actual settlement to preserve auditability.
x402 · MPP · L2 payment channels
⑥ Settlement
Settlement & Escrow
Design the dispute flow before launch, not after first failure. Fund states must be explicit (pending → locked → delivered → accepted/disputed → released/refunded) and delivery proof must be preservable.
ERC-8183 · State machine · Delivery proof
⑦ Identity
Agent Identity
Identity ≠ Trust — it is only the first prerequisite layer. An identity proves who the agent is and who controls it; it does not prove the agent is reliable. Authorization must be explicit and session-scoped, never persistent.
DID/VC · A2A Protocol · ERC-8004
⑧ Trust
Agent Trust & Reputation
Reputation must be task-type-specific, not a single score. An agent excellent at governance summaries may be poor at contract test generation. Credibility comes from verifiable behavior records, not self-declaration.
ERC-8004 · EAS · Attestations

Module Connections at a Glance

From To Relationship Design implication
Chain-aware Context ↔ Web3 Tool Use Bidirectional data flow Tool calls populate context; context requirements drive which tools to invoke
Web3 Tool Use → Agent Workflow Tools called by workflow Workflow orchestrates tool calls; tools must not initiate autonomous actions
Agent Workflow → Agent Wallet Scopes permissions Wallet permissions are defined at workflow design time, not at execution time
Agent Wallet → Machine Payment Executes payments Every payment is a signed transaction from a scoped wallet; no payment without a session key
Machine Payment → Settlement & Escrow Lifecycle completion Payment → escrow lock → delivery check → release or refund; settlement is not a separate system
Agent Identity → Agent Workflow Authorizes (dashed) Workflow steps check identity before allowing high-risk tool calls or payment actions
Agent Identity → Agent Wallet Anchors on-chain identity The wallet address is the agent's primary on-chain identity anchor; same address across transactions
Agent Identity → Agent Trust & Rep Prerequisite Reputation cannot accumulate without a stable, verifiable identity to attach records to
Agent Trust & Rep → Machine Payment Informs counterparty (dashed) Payers check trust score before authorizing auto-payment to an unknown agent endpoint
Settlement & Escrow ↔ Agent Trust & Rep Evidence ↔ credibility Escrow receipts and delivery proofs are the raw material of reputation; high trust can auto-accept delivery
