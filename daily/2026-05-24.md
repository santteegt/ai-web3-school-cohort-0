<!-- Key ideas from chapters; fill in after reading -->

### Frameworks chapter

- Frameworks organize models, tools, state, retrieval, evaluation, and deployment into a maintainable system
- They do not guarantee model correctness, but they can help you split complex systems into clearer modules
- A common mistake is to bend product logic around a framework system boundaries
- Be careful with abstracting too early. It make debugging harder at later stages
- Frameworks must be exit-able: if it makes it hard to change models, vector databases, or deployment methods, the long-term cost will be high.
- A framework is not the only abstraction layer. A model with/out stable tool-calling ability can reduce/increase a lot of parsing and fallback costs
- There are general purpose frameoworks like LangChain, LangGraph, OpenAI Agents SDK, but there are also model / agent ecosystems oriented around tool calling and structured output like Hermes
- Learning agents are systems that can improve from feedback, logs, evaluation results, or user corrections. Such systems should go through an evaluation loop first, then the production system.
- In the AIxWeb3 scenery, frameworks are responsible for connecting model capabilities to product workflows, but should never replace web3 user account permissions, signatures, tx simulation, and account rules.
   - The Product layer should defines user goals, permission boundaries, confirmation flows, and failure handling.
- Chapter notes live is `/knowledge-base/AIxWeb3/raw/Frameworks.md`

### Vibe Coding chapter

- It is a way for humans and AI Coding Agents to iterate software together with the aim to **shortening the engineering feedback loop**.
- **AI coding cannot take over engineering judgment for you**
- A good approach to let a Coding Agent to participate in the engineering process, not unboundedly ask it to rewrite the whole project
- In the AIxWeb3 space, vibe coding can significantly speed up exploration, especially for hackathons and early prototypes, **but chain-related code is higher risk** and must go through review, simulation, and multi-party confirmation
- Chapter notes live is `/knowledge-base/AIxWeb3/raw/Vibe Coding.md`

### MCP Chapter

- MCP aims to abstract the connection layer between the model and external tool into a protocol
  - the client interacts with the model
  - the server exposes **resources, tools, and prompts**
- Tools need properly defined schemas, otherwise tool calls become natural-language parameter guessing
- Different tool calls have different risk levels, so before an MCP server exposes a capability, it should define its permission model
- In the AIxWeb3 space, MCP can be used as the interface layer for Agents connecting to on-chain/developer tools
  - MCP itself is not a wallet-security solution
  - MCP handles tool discovery and call format
  - Web3 account system handles final permission and execution boundaries
- Chapter notes live is `/knowledge-base/AIxWeb3/raw/MCP.md`
