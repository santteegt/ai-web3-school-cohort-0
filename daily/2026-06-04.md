## What I Read / Watched

<!-- Update after today's work -->

- Hallucination isn't solved by better prompts alone
  - it requires external verification: schema validation, rule checks, human confirmation, and audit logs.
  - This is especially important when your AI agent interacts with on-chain systems where fabricated data (bad addresses, wrong balances) can cause real financial harm.
- RAG is not about longer responses — it's about sourced, versioned, and bounded answers
  - For AI × Web3 work, proper citation and freshness tracking prevent hallucination from shifting into the retrieval layer.
- Cobo Agentic Wallet features:
  - [Skill](https://www.cobo.com/products/agentic-wallet/manual/owners/install-skill#install-the-skill-and-create-your-wallet) for agents to create their own wallet
    - Differences between agent using the skill and pairing my own wallet to the agent:
      - When the agent users the skill, it creates a wallet and is immediately the owner with no restrictions.
      - Pairing the agent with the Cobo Agentic Wallet app establishes you as the human controller
  - [MCP Server](https://www.cobo.com/products/agentic-wallet/manual/developer/mcp): if your runtime supports MCP and you want real wallet operations exposed without writing a custom adapter first.
  - CLI, Python & Typescript SDKs for developers
  - Full reference to [Pact policies and completion conditions](https://www.cobo.com/products/agentic-wallet/manual/reference/pact-policies)
- **Excessive Agency** is the vulnerability that *enables damaging actions* to be performed in response to unexpected, ambiguous or manipulated outputs from an LLM, regardless of what is causing the LLM to malfunction
  - Common triggers:
    - *hallucination/confabulation*: by benign prompts or poorly-performant models
    - *direct/indirect prompt injection*: from a malicious user/peer-agent, or compromised extension
  - Root causes:
    - excessive functionality
    - excessive permissions
    - excessive autonomy
  - Prevention/Mitigation strategies:
    - *minimize extensions*: per current intended operation
    - *minimize extension functionality*: e.g. only read-only actions per intended operation
    - *avoid open-ended extensions*: in favour of more granular functionality
    - *minimize extension permissions*
    - *execute extensions in user's context*: track user's authorization and security scope
    - *require user approval*: with human-in-the-loop control for high-impact actions
    - *complete mediation*: enforce authorization/security policies over relying on the LLM to decide which actions are allowed
    - *sanitize LLM inputs/outputs*: follow OWASP’s recommendations; apply SAST, DAST, IAST in development pipelines
  - Recommendations:
    - Log and monitor the activity of LLM extensions and downstream systems to identify where undesirable actions are taking place
    - Implement rate-limiting to reduce the number of undesirable actions that can take place within a given time period
  - NOTE: *Excessive Agency* differs from *Insecure Output Handling*
- **Context Window**: the context window should be used together with **retrieval**, **summarization**, and **structured data strategies**.
- Dispute resolution on settlement & escrow in AIxWeb3 scenarios:
  - Disputes resolve disagreements about delivery qualification — the core boundary between payment and service completion
  - Dispute cost design is a fundamental tradeoff: must prevent abuse without pricing out small tasks
  - Dispute records become critical reputation signals in A2A markets
  - The mechanism requires clear answers: who initiates, what it costs, evidence format, adjudication rules, appeal rights
- Dispute resolution during Escrow
  - Even with crystal-clear standards, a dispute mechanism is still essential as a fallback.
  - **Clear terms reduce disputes but don't eliminate them**: they're a safety valve when acceptance is withheld or delivery is challenged.
