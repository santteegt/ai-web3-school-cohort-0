Template:
```typescript
import { AFCClient } from '@agentfightclub/sdk';
import { CoboClient } from '@cobo/cobo-sdk';

interface TaskProposal {
  id: string;
  proposer: string;
  description: string;
  rewardEth: number;
}

const AFC_MIN_REPUTATION = 10; // threshold

async function handleProposal(proposal: TaskProposal) {
  console.log(`[Agent] Received proposal: ${proposal.id}`);
  
  // Fetch proposer reputation
  const afc = new AFCClient(process.env.AFC_API_KEY);
  const reputation = await afc.getAgentRating(proposal.proposer);
  console.log(`[Agent] ${proposal.proposer} reputation: ${reputation}`);
  
  if (reputation < AFC_MIN_REPUTATION) {
    console.log(`[Agent] Rejected: reputation too low`);
    return { status: 'rejected', reason: 'low_reputation' };
  }
  
  // Sign CAW Pact
  const cobo = new CoboClient({
    apiKey: process.env.COBO_API_KEY,
    apiSecret: process.env.COBO_API_SECRET,
  });
  
  const pact = await cobo.signPact({
    payee: proposal.proposer,
    amount: proposal.rewardEth,
    expiresAt: Math.floor(Date.now() / 1000) + 86400, // 24h
  });
  
  console.log(`[Agent] Signed Pact: ${pact.signature}`);
  
  return {
    status: 'accepted',
    proposalId: proposal.id,
    pactSignature: pact.signature,
    amount: proposal.rewardEth,
  };
}

// Test
const testProposal: TaskProposal = {
  id: 'task-001',
  proposer: '0x...', // your test address
  description: 'Write GuildOS documentation',
  rewardEth: 0.01,
};

handleProposal(testProposal)
  .then(result => console.log('[Agent] Final result:', result))
  .catch(err => console.error('[Agent] Error:', err));
```

## What I read / watched

- Harness Engineering combines prompt engineering (single-turn interactions) and context engineering (multi-turn interaction)  to form a foundational environment where:
  - Agents get the context they need by default
  - Don't require humans' constant supervision every time it executes
  - Effort is migrated from users' repeated labor into the agent system
- AI Agent Intent and Permission Ladder
  1. Code Access: has many good harnesses
  2. Environment Access: still in exploration
  3. Financial Access: large unaddressed
- Financial Harness
  1. Agent is instructed to perform a task that requires external paid API calls
  2. Agent asks for payment approval from the human in the form of a **mandate**
  3. Agent doesn't have to ask for approval for every API call
  4. A risk control layer accesses all the payments under the mandate
- FluxAPay - Mandate as a Sandbox
  - Budget ceiling
  - Intent scope
  - Time window
- Spending validation in Financial Harness
 - Layers: **Rules**, **Intent Matching**, and **Model evaluation**
  - It is harder as a *mistake* may have already settled.
  - Needs validation that operates **before settlement**
  - Model evaluation approach via agent-to-agent supervision
    - When rules can't decide, and intent matching is ambiguous
    - Judgement call decides whether  behaviour is reasonable, hallucinatory, or adversarial
- LLM-as-a-Judge
  - LLM judges excel at open-ended quality evaluation (summaries, citations, completeness)
  - The best approach combines rule-based scoring, LLM judging, and human spot-checks
  - Judge models have their own biases and require calibration over time
  - Using a separate model to judge (not self-evaluation) prevents systematic bias amplification
- Verifiable AI
  - When AI outputs affect assets, permissions, reputation, or governance, **verification cost must match output impact** — turning "believing the model" into "verifying evidence and constraints."
  - Verify source (inputs, model, time) → verify process (environment, replayability) → verify result impact
  - Releasing escrowed funds is a high-stakes action requiring strong verification.
    - Funds release is in the highest-impact tier recommending 'TEE/ZK + multi-evaluator validation
- Reviewed how **Hermes** work and deployed one server instance. Here are some learnings
  - Hermes is an agent that gets better the longer you use it.
  - **Agent-first** architecture (vs Openclaw's gateway-first architecture)
    - everything flows through a single *AIAgent* class in a `run_agent.py` script
    - The core loop is **ReAct-style and synchronous**.
      - Build the system prompt, check if compression is needed
      - make an interruptible API call
      - execute any tool calls
      - and loop again
  - Three separate capabilities sit in one framework:
    - **runtime skill learning**
    - **persistent multi-layer memory**
    - optional **weight-training pipeline**
  - Skills: ~120 bundled + skills.sh + Github taps
  - Memory system - three tier:
    - Bounded markdown
    - FTS5 search
    - 8 external providers
  - The agent has a **hard cap of 90 turns per task**
    - Without it, an agent stuck in a loop (retrying a failing API, re-reading the same file) would silently burn through your credits
  - Layers;
    - **Identity**: `SOUL.md` occupies slot #1 in the system prompt
      - agent's personality, tone, communication, style, and hard limits
    - **Memory**: three layers designed with a specific purpose
      - *Tier 1 - In-Prompt*: injected into the system prompt as a frozen snapshot **when a session starts**. Any updates are not reflected until the next session
        - `MEMORY.md` (2,200 chars max) holds the agent’s notes about your environment, project conventions, tool quirks, and lessons learned.
        - `USER.md` (1,375 chars max) holds your profile: name, communication preferences, skill level, and things to avoid.
        - When > 80% capacity, agent consolidates entries into denser, more information-packed versions
      - *Tier 2 - (full-text) Session search*: every conversation is stored and searchable
        - requires an active search + LLM summarization
      - *Tier 3 - External providers*: 8 pluggable providers that run alongside built-in memory (**never replace it**)
        - Only one can be active at a time
    - **Self-evolving skills**: Markdown+YAML frontmatter and functions as the agent's procedural memory
      - Ships with a **learning loop** that:
        - Remembers across sessions
        - Writes its own reusable skills
        - Prunes them in the background
        - And validates them offline through an evolutionary engine called GEPA
      - The agent creates its own skills autonomously using the `skill_manage` tool. It triggers when:
        - The agent completes a complex task (5+ tool calls)
        - It hits errors or dead ends and finds the working path
        - The user corrects its approach
        - It discovers a non-trivial workflow
      - A **garbage collector** / Curator is a background maintenance system that handles skills inactivity checks
        - Only touches agent-authored skills
        - Idle skills are archived (never deleted) - can be restored later
        - You can pin critical skills via `hermes curator pin <skill>`
    - **GEPA** (Genetic-Pareto Prompt Evolution) for evolving skills
      - Lives in a companion repository (NousResearch/hermes-agent-self-evolution) and operates as an offline optimization pipeline.
      - Proposes targeted improvements through evolutionary search.
